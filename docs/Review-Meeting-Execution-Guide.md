# Silver Guard · 评审会议执行指南

> 版本：v1.0
> 用途：主持人（Chair）/ 记录人（Secretary）/ 全体参会人的行动手册
> 包含：会议准备清单 → 会议引导话术 → 产出物模板 → 归档流程

---

## 一、会议概览

### 两场评审分步安排

| 场次 | 名称 | 主持人 | 时长 | 形式 | 间隔建议 |
| --- | --- | --- | --- | --- | --- |
| 第一场 | **PRD 评审会** | 产品负责人 | **90 分钟** | 线下 / 线上均可 | 间隔 1~2 个工作日 |
| 第二场 | **技术方案评审会** | 技术负责人 | **90 分钟** | 建议线下（有架构图要投影） | 第一场结束后 |

> 如果时间紧张，可将两场合并为一场 **120 分钟**，但建议分开以保证深度讨论。

## 三、第一场：PRD 评审会（90 分钟）

### 3.1 会议流程（主持人版）

```
──────────────────────────────────────────────
开场（5 min）——建立共识
──────────────────────────────────────────────
① 主持人开场（1 min）
   话术："欢迎大家参加 Silver Guard 项目 PRD 评审。
          本次会议目标：锁定 MVP 范围、业务规则、验收标准，
          对齐 3 个 Open Questions，拍板后进入开发。"

② 确认会议规则（2 min）
   - 每条议题讨论不超过 3 分钟，超时由主持人裁定
   - 所有人当场给出结论（通过 / 修改后通过 / 不通过）
   - 记录人实时填写《评审结论》
   - 全程录音（提前说明）

③ 快速回顾 PRD 定位（2 min）
   主持人速过：项目背景 → 目标指标 → MVP 范围
   （投影 PRD 文档第二章）

──────────────────────────────────────────────
范围与边界（20 min）——模块一至三
──────────────────────────────────────────────
④ R-01 ~ R-04（10 min）
   主持人逐条读议题 → 读讨论要点 → 询问各方意见 → 确认结论
   【重点】R-06（30秒 SLA）：强调这是硬性承诺，需要客户/街道书面确认

⑤ R-05 用户故事优先级（5 min）
   建议引导："US-04 / US-05 是否可移至 Phase 2？同意的请举手。"
   快速共识

⑥ R-07 ~ R-08 家属绑定与数据最小化（5 min）
   法务代表重点参与

──────────────────────────────────────────────
业务规则（15 min）——模块三至四
──────────────────────────────────────────────
⑦ R-09 ~ R-13 业务规则（10 min）
   主持人逐条读，网格员/运营代表重点参与

⑧ R-14 ~ R-17 合规与隐私（5 min）
   法务代表必须发言，确认知情同意书与数据脱敏方案

──────────────────────────────────────────────
试点与 Open Questions（20 min）——模块五至六
──────────────────────────────────────────────
⑨ R-18 ~ R-20 试点规模与运营（5 min）

⑩ Open Questions 拍板（15 min）【关键】
   主持人："现在进入最重要的 3 个决策，请各方提前准备的立场。"

   OQ-01：民政/卫健系统 → 现场投票，少数服从多数
   OQ-02：家属端形态 → 小程序方案建议，大多数场景首选
   OQ-03：110/120 联动 → 大多数情况二期，建议明确

──────────────────────────────────────────────
收尾（10 min）
──────────────────────────────────────────────
⑪ 遗留事项整理（5 min）
   记录人当场朗读遗留事项，负责人确认

⑫ 评审结论宣布 + 签字（5 min）
   主持人宣布：通过 / 修改后通过 / 不通过
   所有人签字（纸质或电子签）
```

### 3.2 主持人话术参考

| 场景 | 话术 |
| --- | --- |
| 议题超时 | "这条议题我们先记为'待定'，会后专题讨论。现在进入下一条。" |
| 争议不下 | "我看到两方意见，我先提议：以 R-06 为例，支持 60 秒放宽的请举手，支持 30 秒锁定的请举手。好，我记录为'30 秒锁定，有条件豁免条款'，会后细化。" |
| 有人迟到 | 记录人补签，会议继续，不等待 |
| 偏离议题 | "这个议题很好，但不是今天的范围，我记录为'后续专题讨论'，我们回到 R-XX。" |
| Open Questions 争议激烈 | "我们把 OQ-01 拆成两部分：一期是否打通，打通哪一段？今天先锁定一期方案，二期继续讨论。" |

## 四、第二场：技术方案评审会（90 分钟）

### 4.1 会议流程（主持人版）

```
──────────────────────────────────────────────
开场（5 min）
──────────────────────────────────────────────
① 主持人开场（2 min）
   话术："PRD 评审已完成，今天我们来评审技术方案。
          本次会议目标：确认架构方案、AI 方案、性能基线、安全方案。
          架构已预锁定为单体，Dubbo 通信。"

② 公示 PRD 评审结论（3 min）
   投影 PRD 评审结论 + Open Questions 拍板结果
   告知所有技术决策都以此为基础

──────────────────────────────────────────────
架构与模块（15 min）——模块七
──────────────────────────────────────────────
③ T-01 / T-03 预锁定确认（2 min）
   主持人："T-01 和 T-03 已预锁定为单体 + Dubbo，大家有无异议？无异议视为确认。"

④ T-02 模块归属图评审（13 min）【重点】
   投影 [docs/Module-Ownership-Diagram.md](file:///workspace/docs/Module-Ownership-Diagram.md)
   后端架构师逐模块讲解（5 分钟速讲）
   讨论：各模块负责人确认（M01~M12 负责人明确到人）

──────────────────────────────────────────────
AI / LLM 方案（20 min）——模块八
──────────────────────────────────────────────
⑤ T-04 ~ T-08（20 min）
   AI 负责人重点讲，集体讨论
   【重点】T-06（AI 研判 SLA）：30 秒分解中 AI 占 5 秒，LLM 超时降级策略必须确认

──────────────────────────────────────────────
数据与存储（15 min）——模块九
──────────────────────────────────────────────
⑥ T-09 ~ T-13（15 min）
   后端架构师 + DBA + 运维 共同参与
   【重点】T-12（敏感字段加密）必须与安全代表确认

──────────────────────────────────────────────
接口 / 性能 / 安全 / 部署（25 min）
──────────────────────────────────────────────
⑦ T-14 ~ T-17 接口与集成（8 min）

⑧ T-18 ~ T-21 性能基线（10 min）【关键】
   投影 30 秒 SLA 分解图：
   设备上报（≤1s）→ 网关（≤2s）→ AI 研判（≤5s）→ 通知发送（≤5s）→ 网络余量（≤17s）
   各段负责人确认能否达成

⑨ T-22 ~ T-25 安全方案（7 min）

──────────────────────────────────────────────
部署与运维（10 min）——模块十三
──────────────────────────────────────────────
⑩ T-26 ~ T-30（10 min）
   运维负责人重点讲，确认 Docker Compose 方案与 K8s 时机

──────────────────────────────────────────────
收尾（10 min）
──────────────────────────────────────────────
⑪ 遗留事项整理（5 min）
   记录人当场朗读，负责人确认截止时间

⑫ 评审结论宣布 + 签字（5 min）
```

## 六、评审后归档流程

```
评审结束
  │
  ├─ 当天：记录人发出"会议纪要草稿"至评审群（邮件/飞书/微信）
  │
  ├─ 1 个工作日内：
  │    ├─ 参会人确认纪要内容（24h 内无异议视为通过）
  │    └─ 主持人/记录人将纪要定稿
  │
  ├─ 2 个工作日内：
  │    ├─ 所有签字人完成签字（纸质扫描或电子签）
  │    ├─ 将签字版纪要 + 评审材料（PRD / 架构图）上传至团队共享盘
  │    └─ 发送"评审完成通知"至全体相关人员
  │
  ├─ 3 个工作日内：
  │    ├─ 将评审通过的 PRD 版本更新为 v1.2（冻结）
  │    └─ 遗留事项录入项目管理工具（分配负责人 + 截止日期）
  │
  └─ 评审后 1 周：
       └─ 主持人确认所有遗留事项均已分配，进入开发阶段
```

## 七、常见问题应对

| 问题 | 应对方案 |
| --- | --- |
| 某关键人员（法务/AI负责人）临时缺席 | 会前请其提前在议题清单上填写立场，指定代理人表达意见，结论仍需其会后48h内书面确认 |
| 某议题超时严重，导致后续议题压缩 | 主持人立即宣布该议题为"专题会后讨论"，记录人标记，留待下次专题会 |
| Open Questions 现场无法达成共识 | 主持人提议"暂按方案 A 执行，2 个工作日内由 X 拍板，结果通知全体"，快速推进 |
| PRD 评审不通过，需要大改 | 主持人宣布"评审不通过，需修改后再次评审"，立即明确修改负责人与截止日期，重新安排评审会议 |
| 有人在会上提出新需求（不在 PRD 范围内） | 主持人："好的，这是很好的建议，我记录到'需求池'，下一期规划再讨论。今天我们聚焦 MVP 范围。" |
| 会议超时（超过 90 分钟） | 主持人宣布剩余议题"会后书面评审"，要求所有人 24h 内反馈结论至评审群 |
